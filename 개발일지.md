## 2. 계획 대비 달라진 점 (Deviations)

### 2.1 입력 블록 타입 변경 — 출력 품질 위해 `text` → `fields`
계획서(§4)에서는 일부 섹션을 단일 `text`(멀티라인)로 받기로 했으나, **출력 마크다운이
예시 템플릿의 불릿 구조와 더 일치하도록** 구조화 입력(`fields`)으로 바꿨다.

| 섹션 | 계획 | 실제 | 이유 |
| --- | --- | --- | --- |
| 5. 고객 문제(Pain) | `text` 멀티라인 | `fields` ×4 (문제 1~4) | 항목별 `* **문제 1:** ...` 불릿로 떨어지게 |
| 6. JTBD | `text` | `fields` ×3 | 동일 |
| 9. 신뢰 근거(RTB) | `text` | `fields` ×3 | 동일 |
| 11. 톤 & 키워드 | (분리 고민) | 단일 `fields`(DO/DON'T/키워드) | 섹션 1개 = 블록 1개로 정리해 직렬화 단순화 |

→ 결과적으로 **"섹션 1개 = 블록 1개"** 원칙을 지켜 `buildDoc`의 PART/섹션 헤딩 직렬화가 단순해졌다.

### 2.2 디자인 토큰: branding(라벤더/크림) → TS² (계획 외 추가 요청)
계획서는 자매 프로젝트 `branding`의 라벤더/크림 팔레트를 차용하기로 했다. 구현 직후 사용자가
**TS² 디자인 시스템 적용을 요청**해, `:root` 토큰을 TS² 팔레트(인주 레드 `#AE232A` · 잉크
`#0A0A0A` · 포그 `#9A9A9A` · 애시 `#F4F2EE`)로 리매핑했다. (상세 §3.3)

### 2.3 챕터별 저장 — 계획에 없던 신규 기능 (요청)
계획서의 다운로드는 **전체 1건**(MD/TXT/인쇄)뿐이었다. 사용자가 "수강생 편의를 위해
PART별 저장 + 전체 저장 분리"를 요청해, `buildDoc`/`downloadFile`에 `partIndex` 인자를
추가하고 각 PART 패널 끝에 저장 버튼을 넣었다. (상세 §3.4)

### 2.4 `.gitignore` — 출력예시 폴더 배포 제외로 확정
계획서 주의사항에서 "포함 여부 커밋 단계에서 결정"으로 남겨뒀던 `브랜드맵 최종 출력 예시/`
폴더는 **배포 제외**로 확정(`.gitignore`에 추가). 로컬에서 출력 포맷 기준 자료로만 유지.

### 2.5 문서 산출물이 계획보다 늘어남
계획은 `index.html`/`README.md`/`DEVELOPMENT.md`/`.gitignore` 4종이었다. 진행 중
`기획서.md`(요청)와 이 `개발일지.md`(요청)가 추가됐다. 문서 역할을 다음과 같이 분리:
- `기획서.md` — 왜/무엇(의사결정·정보구조)
- `DEVELOPMENT.md` — 어떻게(아키텍처·배포·확장)
- `개발일지.md` — 무엇이 달라졌나(이 문서)

## 3. 발생한 문제와 해결 (Troubleshooting)

### 3.1 [테스트] 숨은 패널 입력칸에 `page.fill` 타임아웃
**증상**: 챕터별 저장 검증 중 `page.fill('[data-key="bm_positioning"]', ...)`가
`TimeoutError: element is not visible`로 30초 후 실패.
**원인**: `bm_positioning`은 PART 3 패널에 있는데, 탭 UI라 **비활성 패널은 `display:none`**.
Playwright의 `fill`은 보이는 요소만 채운다.
**해결**: 숨은 필드는 DOM API로 직접 값 주입 후 저장 트리거.
```js
el.value = v; saveData();   // page.evaluate 안에서 실행
```
→ 탭형 UI를 헤드리스 검증할 때의 일반 함정. **보이지 않는 탭의 값은 evaluate로 세팅**한다.

### 3.2 [버그 예방] 단일 PART 저장에서 `partIndex===0` falsy 함정
**위험**: `buildDoc(fmt, partIndex)`에서 PART 1의 인덱스는 `0`. `if (partIndex)`로 분기하면
**0이 falsy**라 PART 1 단독 저장이 전체 저장으로 잘못 동작한다.
**해결**: `const single = (partIndex != null)` 로 `null`/`undefined`만 걸러 0을 정상 처리.
`downloadFile`의 파일명 분기도 `if (partIndex != null)`로 통일.
→ 검증에서 PART1 단독 저장이 PART1만 포함(PART2·6 제외)함을 확인.

### 3.3 [스타일] TS² 전환 시 `:root` 밖 하드코딩 색상 누락
**증상**: `:root` 변수만 바꿨는데 일부 색이 여전히 라벤더/퍼플 톤.
**원인**: 변수로 토큰화되지 않은 **하드코딩 색**이 곳곳에 남아 있었다.
- `body::before` 라디얼 그라데이션 `rgba(107,91,138,…)`, `rgba(193,124,90,…)`
- `textarea:focus` / `.keyword-tags input:focus` 포커스 글로우 `rgba(107,91,138,…)`
- `.coaching-note.warn` · `.btn-reset:hover` 배경 `#fdf3ed`(피치)
- `.kakao-btn` 배경 `#211d30`(다크 퍼플)
- `.kakao-code` 텍스트 `#2a2438`(옛 잉크) — **가장 놓치기 쉬웠던 잔여값**
**해결**: `grep -nE "107, ?91, ?138|193, ?124, ?90|42, ?36, ?56|#2a2438|#6b5b8a|#fdf3ed|…"`로
**잔여색을 일괄 스캔**해 전부 TS² 레드/잉크로 교체. 카카오 옐로우 배지(`#FEE500`)만
카카오 브랜드라 유지.
→ 교훈: 디자인 토큰 전환은 `:root` 교체 + **하드코딩 잔여값 grep 스윕**을 한 세트로.

### 3.4 [확장] 전체/단일 직렬화를 한 함수로 — 중복 없이
챕터별 저장을 별도 함수로 만들면 직렬화 로직이 이원화돼 출력이 어긋날 위험.
**해결**: 기존 `buildDoc(fmt)`에 `partIndex` 옵션만 더해
`const list = single ? [WS[partIndex]] : WS`로 순회 대상만 바꿨다. config 단일 소스
원칙 유지 → 전체·단일 출력이 같은 코드에서 파생돼 절대 어긋나지 않음.

### 3.5 [배포] GitHub Pages 첫 빌드 지연 & 404
**증상**: Pages 활성화 직후 라이브 URL이 한동안 `404 Site not found`, 빌드 상태 `building` 지속.
**원인**: 첫 빌드는 수 분 소요. CDN 반영 전까지 404가 정상.
**해결**: 빌드 상태를 폴링하다 `built`가 되면 **캐시버스터 쿼리(`?v=timestamp`)**로 실제
서빙 HTML을 grep 검증. CEO_worksheet에서 겪은 **빌드 큐 직렬 충돌**(진행 중 재빌드 추가
트리거 시 `errored`)을 피하려, 갱신은 push 1회만 하고 완료를 기다렸다.
**관찰**: `pages/builds/latest` API의 commit이 직전 것으로 잠깐 지연 표시될 때도,
실제 서빙 HTML에는 신규 내용이 이미 반영됨 → **최종 판정은 API가 아니라 서빙되는 HTML**로.

### 3.6 [오탐] 스크린샷의 "보라색 알약" — 버그 아님
**증상**: 첫 전체 페이지 스크린샷에서 핵심가치 표 아래 보라색 알약 같은 요소가 보임.
**조사**: 해당 영역을 크롭 확대 → 표·입력칸은 정상. 그 알약은 **full-page 캡처에서 sticky
탭바가 스크롤 위치상 겹쳐 찍힌 것**이었다.
**결론**: 렌더 버그 아님. 콘솔 에러 0, 직렬화 정상으로 교차 확인. → 스크린샷 이상은 곧장
버그로 단정하지 말고 크롭/요소 검증으로 재확인.

### 3.7 [환경] 빈 `.git` 재초기화 안내
**증상**: `git init -b main` 실행 시 "기존 빈 깃 저장소를 다시 초기화했습니다".
**원인**: 프로젝트 폴더에 빈 `.git`이 이미 있었음(이력 없음).
**대응**: 의도한 4개 파일만 스테이징됨을 `git status`로 확인하고 진행. 홈 디렉토리가 통째
git repo인 환경이라(§DEVELOPMENT) **프로젝트 폴더 전용 repo 격리**가 의도대로 유지됨.

## 5. 의사결정 로그 (요약)

- **CSS 변수명 유지, 값만 리매핑**: TS² 전환 시 `--lavender-*` 등 200여 사용처를 리네임하지
  않고 값만 교체. 변수명은 misnomer가 됐지만 변경 위험 최소화. `:root`에 리매핑 주석으로 보완.
- **챕터별 저장은 `.md` 단일 포맷**: 부분 저장은 브랜드맵 본문이 목적이라 MD만. TXT/인쇄는
  전체에만. (요청 시 확장 가능 — 4.x 후속 후보)
- **localStorage 키 분리**(`brandmap_sheet_v1`): CEO_worksheet(`ceo_worksheet_7`)와 충돌 방지.
- **`partIndex != null` 가드**: 인덱스 0 falsy 함정을 구조적으로 차단.
