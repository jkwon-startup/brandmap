## 1. 배경 (Why)

강의 수강생이 브라우저에서 **브랜드맵 워크시트**를 직접 작성하면, `브랜드맵 최종 출력 예시/`
폴더의 완성본(AITY · 스타트업실험실 · TS² 등)과 동일한 구조의 **브랜드맵 마크다운 문서**가
자동 생성되도록 한다. 설치 · 로그인 없이 URL만 공유하면 쓰는 단일 HTML 앱이며 GitHub Pages로 배포한다.

이미 검증된 두 자매 프로젝트가 기반이 됐다.
- `DEV/CEO_worksheet/index.html` — config 단일 소스(`WS` 배열) → 렌더/저장/다운로드 자동 파생,
  localStorage 자동저장, 탭 UI, MD/TXT/인쇄, 진행률, GitHub Pages 배포까지 완성된 **레퍼런스 아키텍처**.
- `DEV/branding/index.html` — 동일 디자인 토큰의 원본 패턴.

→ **새 코드를 발명하지 않고** CEO_worksheet의 아키텍처를 복제하고, 콘텐츠(`WS`)와 출력 직렬화만
**브랜드맵 6 PART / 19 섹션** 구조로 교체하는 것이 핵심 전략이었다.

## 2. 목표 산출물 (출력 포맷)

기준 파일: `브랜드맵 최종 출력 예시/브랜드 맵 예시 .md`

```
# {브랜드명} 브랜드 맵
## PART 1. 브랜드 정의 (The Core Identity)
### 1. 브랜드 정수 (Brand Essence)
...
## PART 6. 미래 계획 (The Future)
### 19. 브랜드 확장 로드맵
```

## 4. 정보 구조 (6 PART / 19 섹션)

| 탭(PART) | 섹션 → 블록 타입 |
| --- | --- |
| PART 1 · 브랜드 정의 | (브랜드명 입력) · 1.브랜드정수 `fields`(Purpose/Vision/Mission) · 2.핵심가치 `table` · 3.브랜드철학 `text` |
| PART 2 · 고객 정의 | 4.핵심고객 `table`(페르소나×3) · 5.고객문제 `fields`(Pain×4) · 6.JTBD `fields`×3 |
| PART 3 · 시장 정의 | 7.경쟁환경 `fields`(직접/간접) · 8.포지셔닝 `text` · 9.RTB `fields`×3 |
| PART 4 · 브랜드 표현 | 10.브랜드개성 `tags` · 11.톤&키워드 `fields` · 12.슬로건 `fields` · 13.감각설계 `fields` |
| PART 5 · 실행 계획 | 14.핵심제품 `text` · 15.주요기능&지표 `table`×5 · 16.고객여정 `fields`(Day1~60+) · 17.제품구성 `fields`(Free/Pro/Ent) |
| PART 6 · 미래 계획 | 18.PDR `fields`(Problem/Disturbance/Resolution) · 19.확장로드맵 `table`(Phase0~3) |

- 각 PART 끝에 `checklist` 블록 → 완료 판정, 진행률 `N/6` 자동.
- `data-key` 규약: `bm_{영역}_{항목}` (예: `bm_purpose`, `bm_val1_name`, `bm_pdr_problem`).

## 5. 기술 설계

| 항목 | 선택 | 이유 |
| --- | --- | --- |
| 구조 | 단일 `index.html` (인라인 CSS+JS) | 빌드 불필요, 정적 호스팅에 그대로 |
| JS | 바닐라 JS | 의존성 0 |
| 외부 의존 | CDN 폰트만 (Noto Serif KR · Pretendard) | 자기완결 |
| 데이터 | localStorage (`brandmap_sheet_v1`) | 서버 불필요 · 무비용 |
| 배포 | GitHub Pages | 무료 · 영구 URL |

**핵심 원칙: config 단일 소스.** `WS` 배열이 렌더링 · 저장 · 다운로드의 단일 진실 소스.

### CEO_worksheet와 다른 점 (이 프로젝트의 차별 구현)
- `ws.part` — 출력 `## PART n` 헤딩.
- `b.skipDoc` — 출력 제외(브랜드명 입력은 제목으로만 사용). `checklist`도 출력 제외.
- 마크다운 헤딩을 `**라벨**` → **`### 라벨`**(섹션 헤딩)으로, `fields`는 `* **필드:** 값`으로 직렬화.
- 문서 제목 `# {bm_name} 브랜드 맵`, 빈 입력은 `(작성 전)`.
- 그 외 렌더 엔진/저장·복원/탭·진행률은 CEO_worksheet와 동일.

> 상세 구현 기록은 [`DEVELOPMENT.md`](./DEVELOPMENT.md) §4 참고.

## 7. 배포 결과

```bash
git init -b main
git add .gitignore index.html README.md DEVELOPMENT.md
git commit -m "init: 브랜드맵 워크시트 웹앱"
gh repo create brandmap --public --source=. --remote=origin --push
gh api -X POST repos/jkwon-startup/brandmap/pages -f "source[branch]=main" -f "source[path]=/"
```

- 라이브 빌드 완료(`built`), HTTP 200, 콘텐츠 무결성 확인.
- 갱신: `git add index.html && git commit -m "..." && git push` (자동 재빌드).
- ⚠️ 빌드 큐는 직렬 — 진행 중 재빌드 추가 트리거 금지(→ `errored` 충돌).

## 9. 향후 확장

- 섹션 문구/예시 변경 → `WS` 배열만 수정 후 push.
- PART/섹션 추가 → `WS`에 객체/블록 추가 (탭 · 진행률 자동 보정).
- 데이터 키(`bm_*`)는 변경 시 기존 작성자 저장값과 끊기므로 유지.
- 기기 간 동기화가 필요해지면 그때 백엔드(Supabase 등) 도입 검토.
